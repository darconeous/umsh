3//! UMSH MAC-layer coordinator and supporting state types.
4//!
5//! > [!NOTE]
6//! > This reference implementation is a work in progress and was developed with
7//! > the assistance of an LLM. It should be considered experimental.
8//!
9//! This crate is the central runtime for the UMSH mesh protocol. It owns every piece of
10//! radio-facing state and drives the full MAC lifecycle: receiving and authenticating
11//! inbound frames, forwarding eligible frames as a repeater, issuing and verifying transport
12//! ACKs, retransmitting unacknowledged sends, suppressing duplicates, enforcing replay
13//! windows, managing frame-counter persistence, and servicing the outbound transmit queue.
14//!
15//! The crate is `no_std` compatible. All data structures are backed by
16//! [`heapless`](https://docs.rs/heapless) fixed-capacity collections; capacity is controlled
17//! by const-generic parameters on [`Mac`] so the compiler enforces sizing at build time with
18//! zero heap allocation.
19//!
20//! # Architecture overview
21//!
22//! ```text
23//! ┌────────────────────────────────────────────────────────────────┐
24//! │  Application / upper layers                                    │
25//! │  queue_broadcast / queue_unicast / queue_multicast / …         │
26//! └──────────────────────────┬─────────────────────────────────────┘
27//!                            │  SendOptions  →  SendReceipt
28//!                            ▼
29//! ┌────────────────────────────────────────────────────────────────┐
30//! │  Mac<P>  (coordinator.rs)                                      │
31//! │                                                                │
32//! │  ┌─────────────────┐  ┌────────────────┐  ┌────────────────┐  │
33//! │  │ IdentitySlot[N] │  │ PeerRegistry   │  │ ChannelTable   │  │
34//! │  │  frame counters │  │  public keys   │  │  channel keys  │  │
35//! │  │  pending ACKs   │  │  cached routes │  │  derived keys  │  │
36//! │  │  pairwise keys  │  └────────────────┘  └────────────────┘  │
37//! │  └─────────────────┘                                           │
38//! │  ┌──────────────────────────────────────────────────────────┐  │
39//! │  │ TxQueue  (priority-ordered outbound frame buffer)        │  │
40//! │  └──────────────────────────────────────────────────────────┘  │
41//! │  ┌──────────────────────────────────────────────────────────┐  │
42//! │  │ DuplicateCache  │  ReplayWindow (per peer, per identity) │  │
43//! │  └──────────────────────────────────────────────────────────┘  │
44//! └──────────────────────────┬─────────────────────────────────────┘
45//!                            │  async next_event()
46//!                            ▼
47//! ┌────────────────────────────────────────────────────────────────┐
48//! │  Platform  (umsh-hal + umsh-crypto)                            │
49//! │  Radio · Clock · Rng · Aes/Sha · CounterStore · KeyValueStore  │
50//! └────────────────────────────────────────────────────────────────┘
51//! ```
52//!
53//! # Modules and key types
54//!
55//! ## [`coordinator`] — the top-level state machine
56//!
57//! [`Mac<P>`] is the single top-level type. Create one with [`Mac::new`], register
58//! identities and peers, then drive it with [`Mac::run`], [`Mac::run_quiet`], or
59//! [`Mac::next_event`] depending on whether you want a long-lived driver loop or manual
60//! multiplexing with other async work. Everything else in this crate exists to support
61//! `Mac`.
62//!
63//! Supporting types in this module:
64//!
65//! - [`LocalIdentityId`] — opaque slot index returned when registering a local keypair.
66//! - [`LocalIdentity`] — either a long-term platform identity or an ephemeral software
67//!   identity for PFS sessions.
68//! - [`IdentitySlot`] — per-identity runtime state: keys, frame counter, pending ACKs.
69//! - [`OperatingPolicy`] — transmission-time rules for the local node (amateur-radio mode,
70//!   operator callsign, per-channel overrides).
71//! - [`RepeaterConfig`] — controls whether and how inbound frames are forwarded.
72//! - [`AmateurRadioMode`] — shared enum governing encryption and identification requirements
73//!   under ham-radio law.
74//! - [`ChannelPolicy`] — per-channel overrides within an [`OperatingPolicy`].
75//! - [`SendError`], [`MacError`], [`CounterPersistenceError`] — error types for queuing,
76//!   runtime event processing, and frame-counter store operations respectively.
77//!
78//! ## [`send`] — outbound transmission types
79//!
80//! - [`SendOptions`] — high-level parameters for a single send: MIC size, encryption,
81//!   flood hops, ACK request, source route, salt, etc.
82//! - [`SendReceipt`] — opaque token returned for ACK-requested sends; matched against
83//!   inbound MAC ACKs to confirm delivery.
84//! - [`TxQueue`] — priority-ordered, fixed-capacity queue of sealed frames waiting for
85//!   radio transmission.
86//! - [`QueuedTx`] — one entry in the transmit queue; includes frame bytes, priority,
87//!   not-before timestamp, and CAD retry count.
88//! - [`TxPriority`] — priority classes from highest (`ImmediateAck`) to lowest
89//!   (`Application`).
90//! - [`AckState`] — ACK lifecycle state machine covering queued sends, forwarding
91//!   confirmation, retry scheduling, and final destination ACK waiting.
92//! - [`PendingAck`] — full tracking record for one in-flight ACK-requested send, stored
93//!   in the identity slot until delivery is confirmed or the deadline expires.
94//! - [`ResendRecord`] — verbatim sealed frame bytes retained for retransmission without
95//!   re-sealing.
96//!
97//! ## [`cache`] — duplicate suppression and replay protection
98//!
99//! - [`DuplicateCache`] — a fixed-size FIFO ring that records recently-seen
100//!   [`DupCacheKey`] values. Before forwarding or delivering any received frame, the
101//!   coordinator checks this cache; matching entries are silently dropped. Prevents
102//!   re-delivery of frames that echoed back via multiple repeater paths.
103//! - [`DupCacheKey`] — keyed on the truncated MIC for authenticated packets (unforgeable
104//!   and compact) or a 32-bit hash of the frame body for unauthenticated ones (broadcast).
105//! - [`ReplayWindow`] — per-peer, per-identity sliding window over frame counters. Rejects
106//!   exact counter replays and frames older than the backtrack window, while tolerating
107//!   a small amount of out-of-order delivery. Backed by a [`RecentMic`] ring for
108//!   backward-window disambiguation.
109//! - [`ReplayVerdict`] — outcome of a replay check: `Accept`, `Duplicate`, or `Replay`.
110//!
111//! ## [`peers`] — remote peer and channel registries
112//!
113//! - [`PeerRegistry`] — a flat list of [`PeerInfo`] records (public key + last-seen time +
114//!   cached route). Looked up by hint or full key when matching inbound packets and routing
115//!   outbound sends.
116//! - [`PeerId`] — opaque index into the peer registry.
117//! - [`CachedRoute`] — either an explicit source route or a flood-distance estimate,
118//!   learned from successfully received packets and used to route future sends without
119//!   flooding.
120//! - [`PeerCryptoMap`] — per-identity map from [`PeerId`] to [`PeerCryptoState`]
121//!   (established pairwise keys + replay window). One map per [`IdentitySlot`].
122//! - [`ChannelTable`] — flat list of registered multicast channels. Each entry stores the
123//!   raw channel key, the derived `k_enc`/`k_mic` keys (precomputed at registration time),
124//!   and the 2-byte channel ID (also precomputed). Looked up by channel ID when
125//!   authenticating inbound multicast and blind-unicast frames.
126//!
127//! ## [`handle`] — shared-ownership coordinator access
128//!
129//! - [`MacHandle`] — a `Copy`-able, lifetime-bounded reference to a `RefCell<Mac<P>>`.
130//!   Designed for multi-task environments (e.g., `tokio` or RTOS task pairs) where one
131//!   task runs the `next_event` loop while another enqueues sends or updates configuration
132//!   without holding a long-lived mutable borrow.
133//!
134//! # Platform trait
135//!
136//! [`Platform`] is the single integration point. Implement it once per deployment target to
137//! supply concrete driver types for all hardware abstractions:
138//!
139//! ```rust,ignore
140//! struct MyPlatform;
141//!
142//! impl umsh_mac::Platform for MyPlatform {
143//!     type Identity = MyHsmIdentity;
144//!     type Aes      = MyAesDriver;
145//!     type Sha      = MyShaDriver;
146//!     type Radio    = MySx1262Driver;
147//!     type Delay    = MyDelay;
148//!     type Clock    = MyMonotonicClock;
149//!     type Rng      = MyTrng;
150//!     type CounterStore = MyFlashStore;
151//!     type KeyValueStore = MyNvmStore;
152//! }
153//! ```
154//!
155//! The `umsh` workspace crate provides a `std`/`tokio`-backed implementation
156//! (`tokio_support::StdPlatform`) suitable for desktop development and testing.
157//!
158//! # Frame-counter persistence
159//!
160//! UMSH uses a monotonic frame counter (not a timestamp) for replay protection. Because the
161//! counter must never reuse a value, it must be committed to non-volatile storage before the
162//! corresponding value is used on-air, or after a power cycle the counter could reset to a
163//! previously-seen value, allowing old ciphertexts to replay. The coordinator manages this
164//! automatically:
165//!
166//! 1. On startup, call [`Mac::load_persisted_counter`] for each long-term identity to read
167//!    the last-committed boundary from the [`umsh_hal::CounterStore`] and set the live
168//!    counter to the next safe starting point.
169//! 2. At runtime, the coordinator schedules a persist whenever the live counter crosses a
170//!    block boundary (every [`COUNTER_PERSIST_BLOCK_SIZE`] frames, default 128). While a
171//!    persist is pending, sends will eventually block with [`SendError::CounterPersistenceLag`]
172//!    if the store is not flushed in time.
173//! 3. The application calls [`Mac::service_counter_persistence`] (typically from the
174//!    `next_event` callback or a background task) to drain the pending write queue.
175//!
176//! # `no_std` usage
177//!
178//! Enable `default-features = false` in `Cargo.toml`. The crate compiles without the
179//! standard library. All capacity limits are compile-time const generics. The `std` feature
180//! enables [`test_support`], which provides software-backed driver stubs for unit testing.

227    type Rng: rand::CryptoRng;

229    type CounterStore: umsh_hal::CounterStore;
230    /// General-purpose persistent key-value store implementation.
231    type KeyValueStore: umsh_hal::KeyValueStore;
232}
