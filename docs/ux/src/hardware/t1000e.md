# SenseCAP T1000-E

The T1000-E is a headless tracker. It has:

- one active-high button;
- one green LED;
- an amplified piezo buzzer;
- a Li-ion battery;
- BLE and LoRa radios;
- a ambient light sensor;
- an accelerometer; and
- critically, no display.

See `docs/t1000e-hardware.md` for electrical details.

Every T1000-E firmware uses the same button recognizer and state model. Button
clicks express action intent through shared action slots. The action assigned to
a configurable slot may depend on the firmware's capabilities; a firmware is
not required to perform an action it cannot support. Mandatory and reserved
slots retain their meaning across firmware flavors.

## State model

The device has three user-facing state variables:

- **Sleep state:** Asleep or Awake
- **Silence state:** Silent or Noisy
- **Battery state:** Battery-Only, Battery-Low, Battery-Critical,
  Battery-Charging, or Battery-Charged

Sleep and Silence are persistent user preferences. Battery State is a mutually
exclusive user-facing summary derived from battery level, external power, and
charger signals. The three state variables are otherwise independent: for
example, the device can be Asleep, Silent, and Battery-Charging at the same
time.

### State precedence

When two states would otherwise request conflicting feedback, apply this order:

1. Asleep suppresses normal application behavior, radio use, the heartbeat,
   and all sounds. Battery-Charging may still use the charging LED pattern.
2. Silent suppresses all sounds while Awake except the two explicit exceptions
   defined under [Silence State](#silence-state).
3. Battery feedback takes priority over the normal heartbeat.
4. While Awake, a short action-confirmation pattern may temporarily replace the
   battery indication, attention indication, or heartbeat, after which the
   underlying pattern resumes.
5. The user-attention indication replaces the normal heartbeat but does not
   replace Battery-Charging or Battery-Low feedback.

### Sleep State

Sleep State is persisted across resets. Asleep is a logical product state, not
one specific MCU power mode: the implementation uses the lowest-power mode that
can preserve the behavior required by Battery State and any permitted USB
interface.

- **Asleep**
  - All radios are off and normal application work is suspended.
  - On battery power, the MCU enters the lowest practical power state, disabling
    as many peripherals as possible.
  - While charging, a low-power monitoring loop may be required to drive the
    charging indication; the device nevertheless remains logically Asleep.
  - The LED is normally off except for the Battery-Charging indication.
  - The buzzer never emits noise.
  - A USB connection to a computer may remain available while external power
    is connected, depending on the firmware flavor. USB availability does not
    by itself make the device Awake.
  - **Transitions**
    - A brief button press clears persisted Sleep State and transitions to
      Awake.
    - A firmware may provide an explicit USB command that transitions to
      Awake. Charger insertion, an ordinary reset, a watchdog reset, or opening
      a USB connection does not implicitly clear Sleep State.
- **Awake**
  - The MCU and radios operate normally according to the current firmware.
  - The LED shows the heartbeat every few seconds when no higher-priority
    pattern is active.
  - If firmware has requested user attention, each heartbeat moment instead
    shows the user-attention pulse defined below.
  - **Transitions**
    - A long button press records Sleep State and transitions to Asleep.
    - A firmware may expose explicit BLE or USB commands that transition to
      Asleep.

Firmware update and recovery may temporarily run regardless of Sleep State,
but must not silently change the stored user preference. Battery-Critical
shutdown records its own reason and does not masquerade as a user-requested
sleep transition.

### Silence State

Silence State is persisted across resets. Silent means the device does not
originate optional sounds. A firmware may expose Silence State through BLE or
USB, but a remotely requested transition emits no confirmation sound.

There are exactly two exceptions to Silent mode:

1. When the user double-clicks to enter Silent, the device emits the brief
   confirmation thump immediately before silence takes effect.
2. An explicit **Find Device** action may sound the buzzer. The user must take
   an unmistakable action whose stated purpose makes it clear that the device
   will emit sound.

No other firmware-specific exception is permitted.

- **Silent**
  - The buzzer does not make sounds except for an explicit Find Device action.
  - **Transitions**
    - While Awake, a double-click enters Noisy and then emits one confirmation
      beep.
- **Noisy**
  - The buzzer may emit confirmation and notification sounds defined by these
    guidelines or by the firmware's documented actions.
  - **Transitions**
    - While Awake, a double-click emits one very short, low-frequency
      confirmation “thump,” then enters Silent. The thump should be as subtle
      as possible while still confirming the transition.

### Battery State

Battery State is a mutually exclusive summary intended to drive user-visible
behavior. Low-battery warnings and critical-battery lockouts apply only while
running from the battery. Connecting external power immediately moves the
device to Battery-Charging or Battery-Charged, so those battery-only behaviors
do not conflict with powered operation.

- **Battery-Only**
  - The device is powered only by its internal battery, which has a normal
    usable charge.
  - No special indication is required beyond ordinary power management.
  - **Transitions**
    - A falling battery level enters Battery-Low.
    - Connecting power enters Battery-Charging or Battery-Charged as
      appropriate. A connection sound may play only while Awake and Noisy.
- **Battery-Low**
  - The battery should be charged soon.
  - A low-battery LED pattern may be shown. An occasional audible warning may
    be used only while Awake and Noisy.
  - **Transitions**
    - Further discharge enters Battery-Critical.
    - Connecting power enters Battery-Charging or Battery-Charged as
      appropriate. A connection sound may play only while Awake and Noisy.
- **Battery-Critical**
  - The battery does not have enough charge to operate the device safely.
  - Normal wake and operation are locked out while external power is absent.
  - **Transitions**
    - Connecting power enters Battery-Charging or Battery-Charged as
      appropriate. No sound is emitted while the device remains Asleep.
- **Battery-Charging**
  - The device is externally powered and the battery is charging.
  - The LED smoothly ramps up and down in a gentle “breathing” pattern, with a
    complete cycle taking roughly 2–4 seconds (0.25–0.5 Hz). This indication is
    permitted while Asleep.
  - Battery-Low warnings and Battery-Critical lockouts are suspended.
  - **Transitions**
    - Charge completion enters Battery-Charged.
    - Disconnecting power enters Battery-Only, Battery-Low, or
      Battery-Critical according to the measured battery level.
- **Battery-Charged**
  - The device is externally powered and the battery is fully charged.
  - While Awake, the heartbeat pulse is slightly longer than normal. While
    Asleep, the LED is off.
  - Battery-Low warnings and Battery-Critical lockouts are suspended.
  - **Transitions**
    - Disconnecting power normally enters Battery-Only. Battery-Low or
      Battery-Critical is allowed if measurement shows an anomalous battery
      condition.

## User attention

A firmware requests **user attention** when it receives a message or another
actionable event that requires user action. The request remains active until
the condition is acknowledged, dismissed, or otherwise cleared by that
firmware.

While Awake in Battery-Only or Battery-Charged, user attention replaces the
ordinary 20 ms heartbeat flash with a smooth brightness ramp up and down lasting
approximately 300 ms. The pulse occurs at the normal heartbeat moments, so it
is more noticeable without increasing the indication frequency or creating a
new blink code for every kind of event.

The attention pulse communicates only “something needs your attention.” The
paired phone or other firmware-specific interface explains what happened and
what action is available. It is not a transmission acknowledgement, progress
indicator, or urgent alarm.

Battery-Charging and Battery-Low retain their higher-priority LED patterns. The
firmware must preserve the pending attention request and begin showing the
attention pulse if those higher-priority conditions clear before the request is
acknowledged. Silence State never suppresses this visual indication.

## Button behavior

When Awake, the shared recognizer maps button sequences to the following action
slots:

| Input | Slot policy | Default or reserved intent |
|---|---|---|
| Single click | Firmware-configurable | Primary safe action; normally send a beacon or check in |
| Double click | Mandatory | Toggle Silence State |
| Triple click | Reserved | Toggle GPS on/off when GPS control is supported |
| Quadruple click | Firmware-configurable | No default action |
| Long press | Mandatory | Transition from Awake to Asleep |

The triple-click slot must not be reassigned to an unrelated action. A firmware
that cannot control GPS leaves it unused. Similarly, any configurable slot that
has no suitable action remains unused.

Short clicks may be held for at most 500 ms; adjacent clicks may be separated
by at most 400 ms. A continuous long press takes priority over an incomplete
multi-click sequence.

Each action that is actually accepted receives an immediate confirmation
indication. In Noisy mode this will typically combine a documented sound with
an LED pattern; in Silent mode the LED provides confirmation. Recognition,
operation completion, and delivery are distinct feedback phases defined by the
higher-level [Status and Feedback](../interaction-model/status-and-feedback.md)
guidelines.

An unused button sequence emits no confirmation. Confirmation indicates that
something happened; the device must not acknowledge an inert slot as though an
action succeeded.

## Stable conventions

This profile establishes the following cross-firmware conventions:

- every firmware uses the same click and hold recognizer;
- single-click is the configurable primary-action slot;
- double-click always controls Silence State;
- triple-click is reserved exclusively for GPS power control;
- quadruple-click is configurable and has no default action;
- a continuous long press always enters Sleep State;
- unsupported action slots remain inert;
- feedback is independent of a phone connection; and
- silence never removes visual feedback; and
- actionable received events use the shared user-attention pulse until cleared.
